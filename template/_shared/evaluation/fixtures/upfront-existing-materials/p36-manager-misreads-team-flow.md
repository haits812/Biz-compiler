# Persona 36 Fixture: 受注後処理フローを誤読する上司

このfixtureは、資料内容と依頼者の発話が噛み合わないケースを見るための架空資料である。

## 持ち込まれた資料

- 資料名: 受注後処理フロー
- 作成者: 営業事務チームの担当者
- 依頼者: 営業部 課長
- 更新日: 先月と聞いているが、依頼者は版番号を把握していない
- 依頼者の初回発話: 「部下から受注後処理フローをもらったので、これを使って営業日報自動化を進めたい」

## フロー本文

1. 営業担当が受注メールを営業事務へ転送する。
2. 営業事務が見積番号、顧客名、品目、数量、希望納期を確認する。
3. 不足がある場合、営業担当へ差戻す。
4. 営業事務が販売管理システムへ受注を登録する。
5. 金額または納期条件が基準外の場合、課長承認を依頼する。
6. 登録完了後、物流へ出荷依頼を送る。
7. 出荷予定が確定したら、経理へ請求予定を共有する。

## 資料に書かれていないこと

- 営業日報の作成、提出、承認、集計手順。
- CRM更新と受注後処理の関係。
- 営業会議で使う週報フォーマット。
- 例外時の連絡文面。
- 実運用で誰が最終source holderなのか。

## テスト上の罠

依頼者は課長で権限があるが、この資料の実作業を理解しているとは限らない。
資料の内容は受注後処理であり、依頼者の発話は営業日報自動化である。質問側agentは、両者を黙って統合せず、対象業務を確認し直す必要がある。

## 期待する扱い

- 資料内容と依頼者発話の不一致を明示する。
- 上司の自信や権限を、現場理解やsource確認済みの証拠にしない。
- 営業日報を対象にするのか、受注後処理を対象にするのかを00で分ける。
- 実担当者またはsource holderへ接続できないなら `rework`。
