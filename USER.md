# USER.md

> このファイルは、Biz-compilerリポジトリで主に話しかけてくる相手の作業モデルを定義する。個人情報の保存場所ではなく、会話・設計・実装の進め方を合わせるための前提である。

## Working Model

- Biz-compilerの構想者・設計者として話している。
- 業務をそのままAIへ丸投げするのではなく、人間、AI、automation、approval gate、external interfaceへ再配分する仕組みを作ろうとしている。
- 粗い発想を会話で固める。最初から全ての仕様が決まっている前提で扱わない。
- 早すぎる結論より、論点分離、構造化、実装可能性、運用時の壊れ方の検査を重視する。
- 用語のニュアンスに敏感で、勝手な脚色や言っていないことの補完を嫌う。
- 未固定領域は怠慢ではなく、上位の判断priorや言語体系が固まっていないために意図的に残されている場合がある。
- 必要なときは、抽象論だけでなく、実際のファイル配置、読み込み順、運用ルールへ落とすことを求める。

## Interaction Notes

- ユーザーが「違う」「それは別」と訂正したら、その区別を以後の前提に入れる。
- ユーザーが出したファイル名、概念名、仮称は、勝手に別の既存概念へ吸収しない。
- 断定する前に、決定済みか、仮説か、保留かを見分ける。
- 会話の中で次回以降も効く制約が出たら、即時メモの対象として扱う。
- フェーズ進行表示は、箇条書きやribbonではなく固定カラムの表を好む。カラムは `#` / `フェーズ` / `この工程の役割` / `今回の扱い` / `完了`。完了マーク `✔` は右端の `完了` カラムで揃える。
