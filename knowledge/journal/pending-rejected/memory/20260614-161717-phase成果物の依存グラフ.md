# phase成果物の依存グラフ / agent-facing phase ledger

## Target
MEMORY.md / knowledge/docs/decisions / future IR schema

## Proposed Change
Biz-compilerの10番台以降のphase成果物は、単なる順序リストではなく、親成果物・入力契約・出力契約・参照元・再検証条件を持つ依存グラフとして扱う候補。完了済みphaseも後続phaseから遡って参照できる必要がある。

特に、ヒアリングした業務を1つずつ自動化していくなら、各数字フェーズで「何を聞いたか」「何を判断したか」「何を凍結したか」「何を成果物にしたか」「どの業務ノードを自動化対象にしたか」を後からagentが遡れる台帳が必須になる。

これは人間向けのKanban UIというより、agent向けの phase ledger / artifact graph / work queue として設計するべき可能性がある。Hermes Kanbanからは parent/child依存、parent results参照、blocked/unblocked、cycle detection、comment log の発想を借りられる。ただしBiz-compilerでは、数字フェーズは作業チケットではなく、業務コンパイルの契約成果物・検証状態・再コンパイル条件を持つ節目として扱う。

## Evidence
ユーザー発話: 10、20...と続けて作業するなら依存関係と完了済み成果物の遡及参照が大切になるはず、Hermesに参考があるか確認したい。

追加発話: Biz-compilerでヒアリングした業務を1個ずつ自動化していくなら、ヒアリング業務の中の各数字をあとから遡って「何をしたか」確認できることが必須。人間向けというよりagent向けのKanbanのようなものが要るかもしれない。

## Risk
早く固定しすぎると、単純な線形pipelineで足りる部分までDAG管理を強制して複雑化する。UI Kanbanを先に作ると、本来必要なartifact ledgerやIR境界が弱くなる。
