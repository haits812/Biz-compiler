# 会話フェーズナビゲーション v0 候補

## Target
future conversation flow / phase ledger / requirements

## Proposed Change
Biz-compilerの初回会話では、ユーザーは基本的に「業務について話しかけてくる」前提でよい。会話開始時に、agentは10-80の工程全体を**カラム付きの表**で提示し、現在位置が10であることを示す。

会話中は、人間向けKanbanではなく、agent-facingなフェーズナビゲーションとして扱う。ユーザーには「今ここ」「次はここ」「この数字で固める成果物」を見せる。agent側は各数字ごとに成果物を凍結し、セッション忘却や後続phaseからの遡及参照に備える。

フェーズ進行表示は、ribbonや縦並びの箇条書きではなく表を基本にする。理由は、状態マークや完了マークが列として揃わないと視認性が悪く、ユーザーが現在地を追いにくいため。フェーズナビゲーション表のカラムは `#` / `フェーズ` / `この工程の役割` / `今回の扱い` / `完了` で固定する。列の追加・削除・並べ替え・ラベル変更をしない。特に `✔` などの完了マークは独立した右端の `完了` カラムに置く。モード差分はカラム変更ではなく、`今回の扱い` の値で表す。

## Draft Flow

1. ユーザーが業務について話し始める。
2. agentは10-80の表を出し、現在phaseを `10: ソース取得` と表示する。
3. 各phase中は、そのphaseの目的、聞くこと、固める成果物、完了条件を短く示す。
4. phase終了時に、agentは次を提示する。
   - このphaseで聞いたこと / 判断したこと
   - 凍結する成果物
   - 未解決論点
   - 次phaseでやること
5. ユーザーが合意したら、Phase Ledgerへ成果物索引と状態を記録し、Event Logへ `phase.freeze` などを追記する。
6. 次phaseへ進むときに、再度表で現在位置と完了済みphaseを表示する。

## Notes

- `00` は必要なら入口判定として内部または短いpreflightで扱う。ユーザー体験上は、業務相談が始まったら基本10から見せてよい。
- フェーズ表は開始時、遷移時、ユーザーが迷った時に出す。表示するなら表形式を維持し、カラムは `#` / `フェーズ` / `この工程の役割` / `今回の扱い` / `完了` に固定する。列を変えない。
- `pipeline-flowchart.html` はこのledgerを可視化する場所であり、phase記録の正本ではない。
- 各数字の成果物は、セッション忘却対策としてPhase LedgerとArtifactに残す。
- agentの都合で表をribbonや箇条書きに変えない。ユーザーの視認性を優先する。
- 80だけ扱う運用セッションでもカラムは変えない。10-60を単に `later` 扱いせず、`今回の扱い` に `参照済み` / `要参照` / `対象外` などを入れ、70/80を `今回やる` として表す。
- `60` は初回構築時の検証、`70/80` は運用後の改善で作り直した成果物の再検証として扱う。検証対象は運用ログそのものではなく、ログやdriftを根拠に作った改善後成果物である。

## Evidence
ユーザー発話: 最初にユーザーが業務について話しかけてくる前提でよい。開始時に10-80まで工程があること、今は10であることを表で示し、10が終わったら次の位置が分かるようにしたい。数字ごとに成果物を固め、セッション忘却対策にもしたい。

追加訂正: 表でないと見にくい。カラムで表を作らないと `✔` が右に寄らず見づらい。agentの自己満足で表示形式を決めない。カラムが毎回ブレるならルールにならないため、フェーズナビゲーション表は `#` / `フェーズ` / `この工程の役割` / `今回の扱い` / `完了` に固定する。状態列は `now/next/later` だけだと意味が薄い。使うなら「今日はやらない」「作った後」など、今回の会話でどう扱うかが分かる用途にする。80だけ扱う場合も列は変えず、`今回の扱い` の値で運用・検証モードを表現する。改善して作った内容の再検証は70/80に含める。

## Risk
毎回大きな表を出すと会話が重くなるが、表示が必要な場面では表形式と固定カラムを崩さない。ユーザー向け表示とagent向けledgerを分けないと、UIが正本化してしまう。
