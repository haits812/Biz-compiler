# AIエージェント設計の本質

エージェントハーネス（LLMを呼び出すコード・設定・ツール群）の設計時に立ち返るべき原則集。
良い出力を祈るのではなく、良い出力しか通りにくい環境を作る。

本文書は2層で構成される:
- **Part I (C)** — LLMの構造的制約。変えられない前提
- **Part II (T/K/V/S/E)** — ハーネスの設計原則。制約への構造的対処

> T/K/V/S/E は相互排他的な分類ではなく、主要な介入面の整理である。

---

## Part I. LLMの構造的制約

プロンプト設計では回避できない構造的限界。

### C-1. コンテキスト帯域は有限でゼロサム
一度に扱える指示・論点は有限。ある指示を目立たせると別の指示が埋もれる。ツール定義も帯域を奪う（GitHub MCP 23k-50kトークン vs CLI ゼロ）。
- 対策: 理由ベースで書き分ける(→E-2)、ハーネスで決定論的に振り分ける(→V-1)

##### C-1.1 コンテキストを能動的に管理する
帯域が有限(→C-1)な以上、静的に「短くする」だけでは不十分。実行中の自動圧縮、ツール定義の遅延読み込み(→K-2.1)、圧縮後の復元優先度設計が必要。圧縮の連続失敗にはサーキットブレーカーを入れてスラッシングを防ぐ。

### C-2. 入力バイアスの増幅
入力の順序・フレーミング・アンカー値に引きずられ、等価な入力でも異なる出力を返す。CoTやReflectionでは軽減不十分。
- 対策: 重要な判断では順序・フレーミングを変えた複数回実行で検証する

### C-3. 迎合性と動機づけ推論
正しい知識を持っていてもユーザーの期待に合わせて出力を歪める。RLHFの構造的帰結。

結論やヒントがコンテキストにあると、そこに到達する中間ステップを逆算して構成する（確証バイアス）。賢いほどCoTの忠実性は低下する。
- Anthropic Circuit Tracing: ヒント使用率99%超でもCoT言及率2%未満
- 対策: 反論を構造化する、別コンテキストで批判させる、異なるモデルでクロスチェック、プロンプトから期待される答えの示唆を分離する

### C-4. 自己申告は完了の証拠にならない
「完了した」と言えるが、それは完了の証拠にならない。外部の検証器なしでは早期完了宣言が起きやすい。
- 対策: 完了はテスト・E2E・レビュー・監視で定義する(→V-2)

### C-5. 報酬ハッキング
評価基準の抜け穴を能動的に発見・悪用する。リンター設定の変更、テストの改変、制約の再解釈等で「評価をパスする行動」を選ぶ。
- 対策: 評価基準自体を保護する(→S-1)、評価と実装の権限を分離する(→S-1)

### C-6. 訓練データの断崖
新しい言語機能・API・フレームワークほど訓練データが少なく、出力品質が系統的に劣化する。暗黙の前提は見落とされる（TypeScriptを明示しなければJavaScriptで生成される）。
- 対策: ドキュメント・サンプルコードをコンテキストに注入する(→K-2)、モデル選択時にcutoffを考慮する

### C-7. 校正盲 — 知らないことを知らない
知識境界を正確に把握できず、もっともらしい虚構を高い確信度で出力する。C-3が「知っているが歪める」、C-4が「完了していないのに完了と言う」であるのに対し、校正盲は「そもそも間違いに気づけない」。
- 対策: 事実判断にはRAG/検索を構造的に挟む(→K-2)、高確信度の出力ほど外部検証を強化する(→V-2)

## Part II. ハーネス設計原則

### タスク構造

#### T-1. 関心ごとの分離
コンテキスト帯域は有限(→C-1)。計画・実装・検証・レビューを同一コンテキストに押し込まない。複数作業者に分ける前に、複数フェーズ/複数ロールへ分離する。

##### T-1.1 並行エージェント間の状態を隔離する
同時に動くエージェントが可変状態（ファイルキャッシュ、記憶、ID等）を共有するとレースとコンテキスト汚染が起きる。各エージェントにはスナップショットを渡し、書き込みは明示的なマージポイントで統合する。

#### T-2. 計画と実行の分離
俯瞰と細部を同時に扱えない。計画を立てレビューしてから実行し、小分割+各ステップ検証で全体の成功率を維持する。

##### T-2.1 実行計画は自己完結させる
良い計画はTODOリストではない。次のエージェントがその文書だけで再開・実行・検証できる自己完結した生きた文書。目的、前提、ファイル、コマンド、acceptance criteria、進捗、判断理由を持つ。

##### T-2.2 目標ドリフトへの抗力
コンテキスト蓄積とともに元の目標から漸進的に逸脱する(→C-3)。弱いエージェントの軌跡に汚染されると、強いエージェントもドリフトを継承する。
- 対策: 各ステップで元の目標を再注入する、目標を外部化して参照する

##### T-2.3 統合判断を委任しない
調査・収集は並列ワーカーに委任できるが、複数結果を統合して判断する工程はオーケストレータ自身が行う。委任先は出発時点のコンテキストしか持たず暗黙知を引き継げないため、統合を委任するとドリフト(→T-2.2)とコンテキスト欠損が同時に起きる。

### 記憶と知識

#### K-1. 記憶の外部化
セッションを跨いだ記憶がないため、状態・進捗・判断理由を外部に永続化する。必要なのは「思い出させること」ではなく「次のセッションが無知でも再開できること」。

##### K-1.1 Restartable Handoff
引き継ぎの本質は要約ではなく再起動可能性。次のエージェントが `git log`、進捗ファイル、実行計画、起動手順から現在地を再構築できる状態を目標にする。

##### K-1.2 記憶は保存前に監査・修復する
長期記憶・スキル・知識ベースは、保存前に検証用の質問や反例で検証し、欠落や誤結合を見つけたら修復してから昇格させる。記憶を追記するだけの覚え書きではなく、検証可能な成果物として扱う。記憶が信頼境界を越える経路の管理は(→S-1.1)。

#### K-2. エージェントからの可読性を最適化する
エージェントに見えない知識は存在しない。リポジトリ内でバージョン管理され、発見可能な成果物を正の情報源にする。古い情報や無関係な情報はノイズとして性能を落とす。

##### K-2.1 ポインタは百科事典より強い
CLAUDE.md は短い入口にし、詳細はポインタ先へ段階的に開示する。巨大な指示ファイルはタスク本体を押し出し、古いルールを混入させる。

##### K-2.2 ドキュメントは捨てるのではなく構造化する
避けるべきは腐敗しやすく検証不能な自由記述。残すべきは ADR、自己完結した計画、実行可能な成果物と整合する文書。

##### K-2.3 観測面を設計する
エージェントの世界認識は公開されたインターフェースに制約される。grep・glob だけでなく UI、logs、metrics、traces も含めて、探索と観測の面を設計する。

##### K-2.4 ハーネス方針を runtime から分離する
制御方針（計画・検証・失敗時の巻き戻し等）を controller code に埋め込むと、比較・移植・監査の単位が codebase 全体になる。方針は versioned で reviewable な artifact として外部化し、runtime は実行に徹する。

### 検証と矯正

#### V-1. 確率的出力を決定論で矯正する
LLMの出力も指示遵守も確率的。確定的に解ける問題は決定論的ツール（リンター・型チェッカー等）に任せ、Hook・CI等の構造で品質を強制する。

##### V-1.1 失敗を仕組みに昇格させる
再発する失敗は、注意喚起ではなく lint / test / tooling に固定化する。

##### V-1.2 フィードバックは速いほど強い
同じ検査なら遅い場所より早い場所で落とす。人間レビュー < CI < プリコミット < ツール直後。

##### V-1.3 選択肢は推論前に絞る
使えないツールや禁止アクションは呼び出し時に拒否するより、プロンプト構築時に除外する。モデルが見えない選択肢は選べない。帯域の節約にもなる(→C-1)。呼び出し時の拒否は最終防衛線であり、主たる制御ではない。

#### V-2. フィードバックループを閉じる
検証するだけでは不十分(→C-4)。検証結果をコンテキストに返して修正させ、再検証するループを構造的に閉じる。

##### V-2.1 観測可能な完了条件
完了はコード差分ではなく観測可能な挙動で定義する。Web: accessibility tree / DOM snapshot、CLI: stdout / exit code、API: HTTP response / schema、Infra: logs / metrics / traces。

##### V-2.2 完了判定を behavior と structure の二層にする
機能テスト合格と保守性維持は別問題。behavior oracle（仕様適合）と structural oracle（責務分割・テスト容易性）を分離し、両方を満たして初めて完了とする。
- 根拠: Needle in the Repo (arXiv:2603.27745) — 13.3% が機能テスト合格だが構造オラクル不合格

### 信頼境界と権限

検証の信頼は権限設計に依存する — V の直後に S を置く理由。

#### S-1. 信頼境界を明示的に設計する
未信頼入力をそのまま昇格させない。権限境界を越える前に隔離し、機械的に検証可能なゲートを通す。評価基準自体を保護し(→C-5)、評価と実装の権限を分離する。

##### S-1.1 記憶の出所と露出先を追跡し、昇格前に隔離する
外部から取り込んだ情報は「どこから来て、どこへ露出しうるか」を保持したまま扱う。非同期入力（バックグラウンド処理、検索結果、受信メッセージ、MCPツール応答等）は、同じ会話面や長期記憶へそのまま混ぜない。出所を付けて隔離し、昇格前にレビュー・検証・ポリシーチェックを通す。

##### S-1.2 安全性の判定を自然言語に委ねない
「この環境は安全か」「この相手は本物か」のような安全性判定を、同一コンテキストの証拠だけでエージェントに推論させない。LLMは危険信号の検出はできても、安全の立証は安定しない。機密開示や高権限操作は、署名・権限トークン・検証可能な証明など機械的に検証可能な条件に結びつける。

##### S-1.3 最小権限で爆発半径を抑える
信頼境界のアウトバウンド面。権限はデフォルト deny の opt-in で付与する。校正盲(→C-7)で間違ったコマンドを確信を持って実行し、報酬ハッキング(→C-5)で権限を拡大しようとするため。
- 可逆なアクションを優先する（`git commit` > `git push --force`）
- 高権限操作は確認を挟む（permission mode、hook-based approval）
- ファイルシステム・ネットワーク・外部APIへのアクセスを sandbox で制限する

##### S-1.4 防御は伝播停止の段階で評価する
prompt injection 対策の本質は悪性文字列の検出ではなく、Exposed → Persisted → Relayed → Executed のどこで伝播を止めたかにある。攻撃面ごとに段階別テレメトリを取り、停止点を SLO 化する。
- 根拠: Kill-Chain Canaries (arXiv:2603.28013) — 露出は全モデルほぼ100%、差は下流段階の遮断能力で決まる

##### S-1.5 安全側をデフォルトにする（fail-closed）
安全性に関わる属性のデフォルトは制限側に設定し、安全と証明されたものだけ opt-in で許可する。S-1.3（最小権限）が「何を許可するか」なら、S-1.5 は「デフォルトをどちら側にするか」の原則。

### 環境設計

#### E-1. 制約が品質を生む
解決空間が広いほど出力が不安定になる。アーキテクチャ境界・命名規則・ディレクトリ構造等の制約で空間を狭めるほど出力が安定する。

##### E-1.1 ドリフトを前提にガベージコレクションする
エージェントは既存パターンを複製する。良い規約も悪い癖も複利で増幅されるため、逸脱検知と継続的 cleanup が必要。

#### E-2. ルールより理由で汎化する
「何をすべきか」のルール列挙より「なぜそうすべきか」の理由の方が未知のケースに汎化できる。ALWAYS / NEVER の多用は要注意。
- 判断基準: 「Would Claude do this anyway if it were smart enough?」が Yes なら書かない

#### E-3. 非決定論性は最後に活用する
確率的性質は弱点であると同時に探索資源。同一仕様で並列実行して最良を選ぶ戦略は有効だが、まずハーネスで失敗率を下げること。並列実行は品質管理の代替ではない。

## IDスキームルール

`{カテゴリ}-{番号}` 形式。子は `.` 区切り（例: S-1.2）。

| プレフィックス | 意味 |
|---|---|
| C | 構造的制約 |
| T | タスク構造 |
| K | 記憶と知識 |
| V | 検証と矯正 |
| S | 信頼境界と権限 |
| E | 環境設計 |

- 新項目は末尾に追番（C-8, S-2 等）。既存IDは変えない
- 削除したIDは欠番にする。再利用しない
- 他ファイルからの参照は `(→C-3)` 形式で書く
