# review-agent-essence

対象のスキル・コード・設計文書を `reference/agent-essence.md` の原則に照らしてレビューし、設計上の強み・弱み・改善案をレビュー・レポートとして返す。

**このスキルは対象を直接修正しない。**

## Input / Output

- **Input**: `$ARGUMENTS` = 対象のパス（スキル、コード、設計文書）
- **Output**: 標準出力にレビュー・レポートを返す。ファイル生成なし
- **評価基準**: `reference/agent-essence.md`

## 手順

### 1. 読み込み

1. `reference/agent-essence.md` を読み、全原則を把握する。
2. `$ARGUMENTS` の対象を読む。対象がSkillなら `SKILL.md` と必要な補助ファイル、設計文書なら関連する参照先まで読む。
3. 対象repoにcoverage policyや既存の検証器がある場合は、docsだけでgap判定せず、実体ファイルを確認する。

### 2. 関連原則の選定

対象の性質に応じて、agent-essenceの原則から関連するものを選定する。全原則を機械的に当てはめない。

選定基準:
- その原則が対象の設計判断に実際に影響するか
- 違反した場合に実害が生じるか
- 適用することで具体的な改善が見込めるか

### 3. 評価

選定した原則ごとに、次を分けて判定する。

| 観点 | 見るもの |
|---|---|
| 関連度 | 高 / 中 / 低 / - |
| 判定 | ○ / △ / × / - |
| 根拠 | 対象ファイル、参照資料、実装、検証器などの具体的な根拠 |
| 影響 | 放置した時の実害、または良い設計判断として効いている点 |

判定「○」にも根拠を書く。良い設計判断の言語化も、このレビューの成果物である。

### 4. レポート出力

以下のフォーマットで、標準出力にレビュー・レポートを返す。

```markdown
# Agent Essence Review Report: {対象名}

> 評価基準: [agent-essence](reference/agent-essence.md)
> 診断対象: {対象パス}

## 対象サマリ

{対象が何をするものか、1-2文で}

## 原則適用サマリ

| 区分 | 件数 | メモ |
|---|---:|---|
| 強く適合 | {N} | {代表的な強み} |
| 改善余地 | {N} | {優先して直す領域} |
| 対象外 | {N} | {対象外にした理由の傾向} |

## 原則適用マトリクス

| # | 原則 | 関連度 | 判定 | 根拠 |
|---|---|---|---|---|
| C-1 | コンテキスト帯域は有限でゼロサム | {高/中/低/-} | {○/△/×/-} | {具体的根拠} |
| T-1 | 関心ごとの分離 | ... | ... | ... |
| K-1 | 記憶の外部化 | ... | ... | ... |
| V-2 | フィードバックループを閉じる | ... | ... | ... |
| S-1 | 信頼境界を明示的に設計する | ... | ... | ... |
| E-2 | ルールより理由で汎化する | ... | ... | ... |

※ 関連度「-」= 対象に無関係、判定も「-」にする。

## 強み

{原則に沿った良い設計判断を具体的に挙げる。ファイルや記述への根拠を添える。}

## 改善提案

{関連度「高」または「中」で判定「△」「×」のものを、影響が大きい順に挙げる。}

### 1. {改善名}

- **関連原則**: {原則IDと名前}
- **現状**: {何が問題か}
- **影響**: {放置した時の実害}
- **提案**: {具体的な改善案}

## 見落としリスク

{対象が暗黙に前提としているが、原則に照らすと危うい設計判断。断定できないものは仮説として書く。}

## 総評

{2-3文で要約。対象の設計成熟度、最優先の改善点、残る未確認事項を述べる。}
```

## Gotchas

- 全原則を均等に扱わない。対象に無関係な原則は「-」にして飛ばす。
- 「原則に書いてないから問題ない」は誤り。原則は網羅的チェックリストではなく思考の補助線である。
- 判定「○」にも根拠を書く。なぜ良いのかの言語化が壁打ちの価値である。
- docsだけでgap判定しない。guard、hook、validator、test、ledgerなど実体がある場合は開いて確認する。
- レポートは出すが、対象ファイルは直接修正しない。修正が必要な場合は提案として返す。
