# ハーネス診断ルーブリック — 25指標

各指標は「パーツの有無」ではなく「制約への構造的対処の一貫性」を問う。
Hookが何件あるかではなく、LLMに任せるべきでないことがLLMに任されていないか。
denyが何件あるかではなく、壊されたら困るものが守られているか。

スコア: ✅(2点) / ⚠ (1点) / ❌(0点) / —(対象外・除外)

> 原則IDはagent-essenceの体系に準拠する（例: →C-1）。

---

## A. 帯域効率 — エージェントの集中力を奪っていないか

### A1. CLAUDE.mdは地図になっているか
**問い**: エージェントは毎セッション、CLAUDE.mdの全内容をコンテキストに抱える。そこに「今このセッションで使わない情報」が混在して、本当に重要な指示の注意力を奪っていないか？

| 判定 | 文脈 |
|------|------|
| ✅ | 毎セッション必要な情報（スタック宣言、基本方針）だけがあり、手順の詳細はポインタで必要時に参照される設計 |
| ⚠ | 一部にベタ書きがあるが、使用頻度が高く実害は小さい |
| ❌ | 低頻度の手順やルールが大量にあり、コンテキストの「机」が散らかっている。エージェントの集中力が分散している |
| — | CLAUDE.mdが存在しない（新規プロジェクト） |

診断の着眼点: 行数自体ではなく「この情報は本当に毎セッション必要か？」を各セクションに問う。200行でも全て毎回参照されるなら正当、30行でも月1回しか使わない手順があるなら帯域の無駄
関連原則: (→C-1) コンテキスト帯域は有限でゼロサム, (→K-2.1) ポインタは百科事典より強い

### A2. ツール接続は帯域コストに見合っているか
**問い**: MCP接続はツール定義がコンテキストに常駐する。そのコストに見合う価値があるか？CLIで同等のことができるのに帯域を払っていないか？

| 判定 | 文脈 |
|------|------|
| ✅ | 各MCP接続にCLIでは代替できない理由がある（ステートフル接続、CLI未提供等）。帯域コストと価値が釣り合っている |
| ⚠ | CLI代替が可能だが、環境制約（認証、インストール不可等）で正当な選択 |
| ❌ | `gh`, `curl`, `jq` 等のCLIで同等の機能が帯域ゼロで実現できるのに、MCPのツール定義が帯域を消費し続けている |
| — | MCP未使用 |

診断の着眼点: MCP接続ごとに「これはCLIで代替できないか？」を個別に問う。接続数ではなく、各接続の必然性
関連原則: (→C-1) CLIは帯域ゼロで実行できる

### A3. Claudeが既に知っていることを教え直していないか
**問い**: スキルやCLAUDE.mdに書かれた知識は、Claudeの訓練データに存在しない情報か？Claudeが自力で正確に対応できることを、わざわざ帯域を使って再記述していないか？

| 判定 | 文脈 |
|------|------|
| ✅ | 自作CLI、社内ツール、cutoff後API、プロジェクト固有の規約など、Claudeが知りえない知識を補っている |
| ⚠ | メジャーFWだが、プロジェクト固有のバージョンや使い方に特化した情報がある |
| ❌ | git, npm, React等の基本操作をスキルやCLAUDE.mdで再記述している。Claudeは既に知っている |
| — | リファレンス目的のスキルやドキュメントが存在しない |

診断の着眼点: 各リファレンス情報に「Claudeにこれなしで同じ作業を頼んだら、間違えるか？」を問う。Yesなら必要、Noなら帯域の浪費
関連原則: (→C-6) 訓練データの断崖, (→C-7) 校正盲, (→E-2) 賢いClaudeなら言われなくても分かること

### A4. エージェントの選択肢が不必要に広がっていないか
**問い**: スキルのdescription一覧、permissions.allow、ツール定義 — これらはエージェントの「メニュー」。メニューが長いほど注文に迷う。使わないメニューが並んでいないか？

| 判定 | 文脈 |
|------|------|
| ✅ | リファレンス専用スキルは自動発動を抑制。allowは必要最小限。エージェントの視界に不要な選択肢が入らない設計 |
| ⚠ | 一部に過剰な選択肢があるが、実害は限定的 |
| ❌ | 全スキルが自動発動可能、またはallowがワイルドカードで実質全許可。エージェントの判断リソースが無駄に消費されている |
| — | スキル5個未満 かつ allow未設定で、選択肢の圧迫がない |

診断の着眼点: 「このスキル/許可は、エージェントが毎回の判断で考慮する必要があるか？」を問う。Noなら `disable-model-invocation: true` や allow の絞り込みで視界から外す
関連原則: (→V-1.3) 選択肢は推論前に絞る, (→C-1.1) コンテキストの能動的管理, (→S-1.5) 安全側をデフォルトにする

### A5. 長いセッションへの備えがあるか
**問い**: コンテキストは使い続けると枯渇する。重要な判断理由や計画がコンテキスト内の会話にしか存在しない場合、圧縮で失われる。セッションが長くなっても情報が生き残る設計があるか？

| 判定 | 文脈 |
|------|------|
| ✅ | 重要な判断理由・計画・進捗がファイルに外部化されている。圧縮されても `Read` で復元可能 |
| ⚠ | 一部は外部化されているが、体系的ではない |
| ❌ | 全てが会話内にのみ存在。長いセッションで判断理由が消えても復元手段がない |
| — | 短いセッションのみの利用で、枯渇リスクが実質ない |

診断の着眼点: 「今このセッションが途中で圧縮されたら、何が失われるか？」を問う。失われて困るものがファイルに外部化されているか
関連原則: (→C-1.1) コンテキストの能動的管理, (→K-1) 記憶の外部化

## B. 検証の堅牢性 — 「完了」を誰が保証しているか

### B1. LLMに判断させるべきでないことをLLMに任せていないか
**問い**: 構文エラー、フォーマット崩れ、型不整合 — これらは正解が一意に決まる。LLMの「たぶん大丈夫」ではなく、決定論的なツール（lint、型チェッカー、フォーマッター）で判定すべきもの。それが自動化されているか？

| 判定 | 文脈 |
|------|------|
| ✅ | 正解が一意に決まるルールがHook・lint・CI等の決定論的ツールで自動検証されている（settings.jsonのHook、プラグイン経由、CI等の手段は問わない） |
| ⚠ | 一部は自動化されているが、カバー範囲に穴がある |
| ❌ | 確定的に判定できるルールが全てLLMの自主性に委ねられている。「気をつけて」とプロンプトで頼んでいるだけ |
| — | プロジェクトに確定的ルールの対象がない |

診断の着眼点: 「このプロジェクトで、正解が一意に決まるルールは何か？それは決定論的に検証されているか？」を問う。手段（Hook/CI/プラグイン）ではなく、カバーされているかどうか
関連原則: (→V-1) 確率的出力を決定論で矯正する, (→V-1.2) フィードバックは速いほど強い

### B2. 「完了しました」を鵜呑みにしていないか
**問い**: LLMの「完了しました」は完了の証拠にならない。機能面（テストが通る、APIが応答する）と構造面（保守性、責務分離）の両方で、外部から検証する仕組みがあるか？

| 判定 | 文脈 |
|------|------|
| ✅ | 機能検証（exit code、HTTP response等）と構造検証（lint、レビュー等）の両方が組み込まれている |
| ⚠ | 片方はあるが、もう片方が欠けている |
| ❌ | 完了判定が全てLLMの自己申告。外部から検証する仕組みがない |
| — | 探索的タスクが中心で、機械的な完了定義が構造的に困難 |

診断の着眼点: 「エージェントが『完了しました』と言ったとき、それを信じる以外の根拠があるか？」を問う
関連原則: (→C-4) 自己申告は完了の証拠にならない, (→V-2.1) 観測可能な完了条件, (→V-2.2) behavior+structure二層完了判定

### B3. 検証結果は修正に繋がっているか
**問い**: 検証するだけでは不十分。検証で見つかった問題がコンテキストに返されて修正され、再検証されるループが閉じているか？

| 判定 | 文脈 |
|------|------|
| ✅ | 検証→修正→再検証のループがスキル・Hook・プラグイン等で構造的に閉じている |
| ⚠ | ユーザーが手動で介在してループを回している。機能はするが構造化されていない |
| ❌ | 検証はあるが結果が放置される。lintエラーを出すが修正に接続されていない |
| — | 単発タスクでループが不要 |

診断の着眼点: 「検証で問題が見つかったとき、次に何が起きるか？」を問う。人間が気づいて指示しないと修正されないなら、ループは開いている
関連原則: (→V-2) フィードバックループを閉じる

### B4. 問題はできるだけ早く見つかるか
**問い**: 同じ問題でも、ツール実行直後に見つかるのと、CIで見つかるのと、人間のレビューで見つかるのでは、手戻りコストが桁違い。フィードバックの速度を意識した設計になっているか？

| 判定 | 文脈 |
|------|------|
| ✅ | ツール実行直後（PostToolUse Hook等）で即時検証。問題が小さいうちに捕まえている |
| ⚠ | CIやPreCommitで検証しているが、ツール直後の即時フィードバックはない |
| ❌ | 問題が見つかるのはCI失敗時かユーザーの目視レビューのみ。手戻りが大きい |
| — | 検証対象がない |

診断の着眼点: 「問題が見つかる最も早いタイミングはいつか？もっと前倒しできないか？」を問う
関連原則: (→V-1.2) ツール直後 > プリコミット > CI > 人間レビュー

### B5. 同じ失敗を二度プロンプトで注意していないか
**問い**: 同じ失敗が繰り返されたとき、CLAUDE.mdに「〜しないでください」と書くのは対症療法。LLMは「注意」を忘れる。失敗パターンをHook・lint・testに固定化して、構造的に再発不能にしているか？

| 判定 | 文脈 |
|------|------|
| ✅ | 過去の失敗パターンがHook・lint・test等の決定論的な仕組みに昇格済み。プロンプトで注意する必要がない |
| ⚠ | CLAUDE.mdに「〜しないで」が少数残っているが、仕組み化の余地を認識している |
| ❌ | 「注意」「禁止」「NEVER」が多数あり、全て自然言語の注意喚起。同じ失敗が起きるかどうかはLLMの自主性次第 |
| — | プロジェクトが新しく、再発パターンがまだない |

診断の着眼点: CLAUDE.mdの「〜しないで」「NEVER」「ALWAYS」を数える。それぞれについて「これはHookやlintで強制できないか？」を問う。仕組み化できるのにプロンプトに留まっているなら、昇格の余地がある
関連原則: (→V-1.1) 失敗を仕組みに昇格させる

## C. 権限と信頼境界 — 善意の間違いの爆発半径は抑えられているか

### C1. 品質のものさし自体が守られているか
**問い**: テスト、lint設定、CI設定 — これらは品質を測る「ものさし」。エージェントがものさしを書き換えられるなら、どんな測定結果も信用できない。ものさしはエージェントの手の届かない場所にあるか？

| 判定 | 文脈 |
|------|------|
| ✅ | テスト・lint設定・CI設定等がpermissions.denyで保護されている。エージェントは「ものさし」を変更できない |
| ⚠ | 一部は保護されているが、漏れがある |
| ❌ | 品質の評価基準がエージェントの編集可能領域にある。テストを書き換えてテストを通せる状態 |
| — | テスト/lint設定が存在しないプロジェクト |

診断の着眼点: 「エージェントが"テスト全通過"を報告したとき、テスト自体が改変されていない保証があるか？」を問う
関連原則: (→C-5) 報酬ハッキング, (→S-1) 信頼境界を明示的に設計する

### C2. 取り返しのつかない操作が気軽にできない設計か
**問い**: `push --force`, `rm -rf`, `reset --hard` — これらは一瞬で取り返しのつかない結果を生む。エージェントは間違いを確信を持って実行する（→C-7）。不可逆操作に対して、ガードレールがあるか？

| 判定 | 文脈 |
|------|------|
| ✅ | 不可逆操作がdenyまたはHookで構造的にブロックされている |
| ⚠ | 一部は制限されているが、網羅的ではない |
| ❌ | 不可逆操作が制限なし。善意の間違いで取り返しのつかない損害が起きうる |
| — | 個人実験プロジェクトで爆発半径が極小（ただしリスクは指摘する） |

診断の着眼点: 「エージェントが誤って `rm -rf` や `push --force` を実行したら、何が失われるか？その被害を防ぐ仕組みがあるか？」を問う
関連原則: (→S-1.3) 最小権限で爆発半径を抑える, (→C-7) 校正盲

### C3. エージェントが自分のルールを書き換えられないか
**問い**: permissions.denyは「エージェントにできないこと」を定義する。しかしエージェントがsettings.json自体を編集できるなら、自分で自分のdenyリストを消せる。鍵のかかった金庫の鍵が机の上に置いてあるようなもの。

| 判定 | 文脈 |
|------|------|
| ✅ | settings.jsonがプロジェクト・ユーザーの両レベルでpermissions.denyに含まれている |
| ⚠ | 片方のレベルは保護されているが、もう片方が未保護 |
| ❌ | settings.jsonの編集が可能。エージェントが自分のdenyリストを解除できる |

診断の着眼点: 「denyリストを追加しても、エージェントがそのdenyを消せるなら意味があるか？」を問う
関連原則: (→C-5) 報酬ハッキング, (→S-1) 評価と実装の権限分離

### C4. 外の世界に影響する操作に歯止めがあるか
**問い**: git push、PR作成、deploy、メッセージ送信 — これらはローカル環境を超えて外の世界に影響する。ローカルの間違いはやり直せるが、外に出た間違いは取り消せないことがある。確認ステップがあるか？

| 判定 | 文脈 |
|------|------|
| ✅ | 外部影響操作にHook・Permission Modeによる確認が設計されている |
| ⚠ | Permission Modeのデフォルト確認に依存。意図的な設計ではないが機能はしている |
| ❌ | 外部影響操作がallowで自動実行される設定。確認なしで外の世界に影響を与えられる |
| — | 外部サービスとの連携がないプロジェクト |

診断の着眼点: 「エージェントが誤って本番にpushしたら、それを止める仕組みがあるか？」を問う。意図的なallow（CLAUDE.mdで「自動push」と明記）は正当、無自覚なallowは危険
関連原則: (→S-1.3) 最小権限, (→S-1.5) 安全側をデフォルトにする

### C5. 外から入ってくる情報を疑っているか
**問い**: MCP応答、WebSearch結果、受信メッセージ — 外部から入る情報はエージェントの行動を変える。信頼できる情報と不確実な情報を区別せず混ぜていないか？

| 判定 | 文脈 |
|------|------|
| ✅ | 外部入力を隔離・検証してから使う設計がある（subagentでの検疫、出所タグ付け等） |
| ⚠ | 特別な隔離はしていないが、信頼できるソースのみに接続を限定している |
| ❌ | 不特定の外部入力をそのまま長期記憶やプロンプトに注入。注入攻撃のリスク |
| — | 外部入力ソースがない（CLI・ローカルファイルのみ） |

診断の着眼点: 「外部から入った情報がそのままMemoryに保存されたり、次の判断に影響したりする経路はないか？」を問う
関連原則: (→S-1.1) 記憶の出所と露出先を追跡, (→S-1.2) 安全性の判定を自然言語に委ねない, (→S-1.4) 防御は伝播停止の段階で評価する

## D. 知識と記憶 — エージェントは必要なことを知っていて、不要なことに惑わされていないか

### D1. 暗黙の前提に頼っていないか
**問い**: 技術スタックを明示しないと、Claudeは訓練データで最も頻出する選択肢をデフォルトにする（Vitest→Jest, pnpm→npm）。「言わなくても分かるだろう」が品質のバラつきを生んでいないか？

| 判定 | 文脈 |
|------|------|
| ✅ | 言語・FW・ツールが明示宣言されている。非主流の選択には「〜ではない」の排除宣言がある |
| ⚠ | 主要技術は明示だが、非主流選択の排除宣言がない |
| ❌ | スタック宣言がない。Claudeが毎セッション推測している |
| — | 技術選択の余地がないプロジェクト |

診断の着眼点: 「新しいセッションでClaudeがこのプロジェクトに初めて触れたとき、どの技術を使うべきか迷わないか？」を問う
関連原則: (→C-6) 訓練データの断崖, (→C-2) 暗黙のデフォルトは入力バイアスの一種

### D2. 嘘を教えていないか
**問い**: CLAUDE.mdに書かれたパス、コマンド、ルールが実態と一致しないと、エージェントは存在しないファイルを参照し、動かないコマンドを実行する。情報が古くなっていないか？

| 判定 | 文脈 |
|------|------|
| ✅ | CLAUDE.mdの記述が現在のファイル構造・設定と一致している |
| ⚠ | 一部に古い記述があるが、致命的ではない |
| ❌ | 存在しないディレクトリ・ファイル・コマンドへの言及がある |
| — | CLAUDE.mdが存在しない |

診断の着眼点: CLAUDE.md内のパス参照を実際にglob/grepで確認する。「エージェントがこの記述を信じて行動したら、正しく動くか？」を問う
関連原則: (→K-2) 古い情報はノイズ, (→K-2.2) ドキュメントは構造化する

### D3. セッション間の記憶が腐っていないか
**問い**: Memoryはセッション間の引き継ぎ手段。整理されていなければノイズになり、古ければ嘘になる。次のセッションが、このMemoryだけで正しく再開できるか？

| 判定 | 文脈 |
|------|------|
| ✅ | インデックスがあり、カテゴリ分類され、各エントリが現在の状態を反映している |
| ⚠ | 使っているが、分類不十分または一部に古いエントリがある |
| ❌ | 未整理で大量のエントリが蓄積。または長期間メンテされていない |
| — | Memory機能を使っていない（新規ユーザー） |

診断の着眼点: 「このMemoryを読んだ新しいセッションのClaudeは、正しい判断ができるか？誤った記憶に基づいて間違えないか？」を問う
関連原則: (→K-1) 記憶の外部化, (→K-1.1) Restartable Handoff, (→K-1.2) 記憶は保存前に監査・修復する

### D4. Claudeが知らない技術に丸腰で挑んでいないか
**問い**: 訓練データのcutoff後の技術や自作ツールについて、Claudeは「知らない」と言わずに自信満々に間違える（校正盲）。知識の穴を埋めるリファレンスが用意されているか？

| 判定 | 文脈 |
|------|------|
| ✅ | cutoff後技術・自作ツールのリファレンスが整備されている |
| ⚠ | 一部にリファレンスがあるが、網羅的ではない |
| ❌ | cutoff後技術を使っているのにリファレンスがない。Claudeが推測で補っている |
| — | cutoff後技術を使っていない |

診断の着眼点: 「このプロジェクトで使っている技術のうち、Claudeの訓練データに十分含まれていないものは何か？それに対するリファレンスがあるか？」を問う
関連原則: (→C-6) ランタイム辞書パターン, (→C-7) 校正盲

### D5. 暗黙知がエージェントから見えない場所に隠れていないか
**問い**: エージェントの世界はgrep/glob/Readで到達できるファイルに限られる。頭の中にしかない規約、口頭伝達のルール、コードのコメントに埋もれた方針 — これらはエージェントにとって存在しない。

| 判定 | 文脈 |
|------|------|
| ✅ | 重要な規約がCLAUDE.md・docs・スキル内に構造化され、検索で発見可能 |
| ⚠ | 主要な規約は発見可能だが、一部が散在 |
| ❌ | 重要な規約がリポジトリ内に存在しない。暗黙知に依存している |
| — | 規約が必要ない規模 |

診断の着眼点: 「このプロジェクトの重要なルールで、CLAUDE.mdにもdocsにもスキルにも書かれていないものはないか？」を問う
関連原則: (→K-2) エージェントからの可読性, (→K-2.3) 観測面を設計する, (→K-2.4) ハーネス方針をruntimeから分離する

## E. 環境設計 — 良い出力しか出にくい環境が作れているか

### E1. 指示が未知の状況に対応できるか
**問い**: 「常にXせよ」「絶対にYするな」は想定済みの状況でしか機能しない。理由（なぜXすべきか）が書かれていれば、想定外の状況でもエージェントが判断できる。また、「いい感じにレビューして」のような曖昧な指示は迎合を誘発する。指示の質が品質を決めていないか？

| 判定 | 文脈 |
|------|------|
| ✅ | 指示に理由が付いている。迎合誘発表現がない。未知のケースに汎化できる |
| ⚠ | 理由付き指示が主だが、一部にルール列挙やアンカー表現がある |
| ❌ | 理由なしの命令形が支配的。または迎合誘発表現がレビュー指示に含まれている |
| — | CLAUDE.mdの指示が少なく、判定対象が不十分 |

診断の着眼点: 「CLAUDE.mdの各指示について、想定外の状況が来たとき、エージェントは正しく判断できるか？理由が書いてあるか？」を問う。「ALWAYS」「NEVER」の数を数えるのではなく、理由の有無を見る
関連原則: (→E-2) ルールより理由で汎化する, (→C-2) 入力バイアス, (→C-3) 迎合性

### E2. 出力がバラつく余地を構造で抑えているか
**問い**: 命名規則、ディレクトリ構造、参照実装 — これらは「正解の範囲」を狭める制約。制約がないとエージェントは毎回違うスタイルで書く。制約が品質を生んでいるか？

| 判定 | 文脈 |
|------|------|
| ✅ | 命名規則・ディレクトリ構造・参照実装が明記され、出力が安定している |
| ⚠ | 一部の制約はあるが、広い範囲で自由度が残っている |
| ❌ | 制約がなく、エージェントが毎回異なる構造を選ぶ |
| — | プロジェクト初期で制約を定めるのが時期尚早 |

診断の着眼点: 「同じタスクを2回頼んだら、同じ構造の出力が返ってくるか？バラつくなら、何の制約が足りないか？」を問う
関連原則: (→E-1) 制約が品質を生む

### E3. 自分の作品を自分で採点していないか
**問い**: 生成と評価を同じコンテキストで行うと、迎合性が働いて自分の出力に甘くなる。品質が重要なフローで、生成者と評価者が物理的に分離されているか？

| 判定 | 文脈 |
|------|------|
| ✅ | 生成と評価が `context: fork` で別コンテキストに分離されている |
| ⚠ | 分離を意識しているが、一部で混在 |
| ❌ | 全フローで同一コンテキスト内で生成→自己レビュー |
| — | レビューが不要な単純タスクのみ |

診断の着眼点: 「レビュー結果を出すエージェントは、レビュー対象を作ったエージェントと同じコンテキストか？同じなら、そのレビューは信用できるか？」を問う
関連原則: (→C-3) 迎合性, (→T-1) 関心ごとの分離, (→T-1.1) 並行エージェント間の状態を隔離する

### E4. 時間とともに劣化する力学に対抗しているか
**問い**: エージェントは既存パターンを複製する。良い規約も悪い癖も増幅される。古いGotchasが残り、目標がドリフトし、ルールが陳腐化する。この自然な劣化に対抗する仕組みがあるか？

| 判定 | 文脈 |
|------|------|
| ✅ | 逸脱検知の仕組みがある。ワークフローの各ステップで目標を再注入している。Gotchasの定期見直しが行われている |
| ⚠ | CLAUDE.mdの更新で対応しているが、体系的な仕組みではない |
| ❌ | 逸脱検知なし。古いルールが蓄積。長いワークフローで元の目標が忘れられる |
| — | プロジェクトが新しく、劣化が蓄積する期間が経っていない |

診断の着眼点: 「3ヶ月後のこのハーネスは、今より良くなっているか、悪くなっているか？劣化を防ぐ仕組みがあるか？」を問う
関連原則: (→E-1.1) ドリフトを前提にガベージコレクションする, (→T-2.2) 目標ドリフトへの抗力

### E5. 複雑なタスクを分割して、各段階で品質を担保しているか
**問い**: 「全部やって」は自然な衝動だが、10ステップの成功率が各90%でも全体は35%。分割して各段階で検証しないと、全体の品質は掛け算で劣化する。タスク構造が品質を支えているか？

| 判定 | 文脈 |
|------|------|
| ✅ | 複雑なタスクがフェーズに分離され、計画に目的・前提・acceptance criteriaがある。統合判断は親コンテキストが行い、下位に委任していない |
| ⚠ | 一部で分離があるが、計画が自己完結していない、または統合判断をsubagentに委任している |
| ❌ | 全てを1プロンプトで処理。計画も実装も検証も同一コンテキスト |
| — | タスク分離が不要な規模 |

診断の着眼点: 「このプロジェクトで最も複雑なタスクは何か？それは分割されているか？各段階にゲートがあるか？」を問う
関連原則: (→T-1) 関心ごとの分離, (→T-2) 計画と実行の分離, (→T-2.1) 実行計画は自己完結させる, (→T-2.3) 統合判断を委任しない
