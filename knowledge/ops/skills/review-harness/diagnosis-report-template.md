## Quick Wins リファレンス例

よくある初期状態に対する具体的な改善例。テンプレート使用時にこれらを参考にコピペ可能な提案を作成する。

### 例1. 評価基準の保護（C1対応、5分）

settings.jsonが未設定、またはdenyが空の場合に最初に提案する。

```jsonc
// .claude/settings.json に追加（なければ新規作成）
{
  "permissions": {
    "deny": [
      "Edit(.eslintrc*)",
      "Edit(biome.json)",
      "Edit(**/*.test.ts)",
      "Edit(**/*.spec.ts)",
      "Edit(tsconfig.json)",
      "Edit(.claude/settings*)"
    ]
  }
}
```

### 例2. 最初のPostToolUse Hook（B1対応、10分）

Hook未設定の場合。Write/Edit後に自動でフォーマッターを実行する例。

```jsonc
// .claude/settings.json の hooks セクションに追加
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "npx biome check --write \"$TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
      }
    ]
  }
}
```

### 例3. CLAUDE.mdのスタック宣言（D1対応、5分）

技術スタックが明示されていない場合。

```markdown
## Stack
- 言語: TypeScript (strict mode)
- テストFW: Vitest（Jestではない）
- パッケージマネージャ: pnpm（npmやyarnではない）
- CSS: Tailwind CSS v4
```

### 例4. 不可逆操作の制限（C2対応、5分）

```jsonc
// .claude/settings.json の permissions.deny に追加
{
  "permissions": {
    "deny": [
      "Bash(git push --force*)",
      "Bash(git reset --hard*)",
      "Bash(rm -rf *)"
    ]
  }
}
```

### 例5. CLAUDE.mdのポインタ化（A1対応、30分）

100行超のCLAUDE.mdを圧縮する場合。

```markdown
## Testing
docs/testing.md を参照。`npm test` で実行

## Architecture
docs/architecture.md を参照

## Deploy
docs/deploy.md を参照
```

## テンプレート使用時の注意

- **検出されたアンチパターン**: 優先順位（C>B>A>D>E）に従い記述。同カテゴリ内では❌を先に
- **Quick Wins**: 上記リファレンス例を参考に、対象プロジェクトに合わせてカスタマイズする。必ずコピペ可能にする
- **総評**: 「スコアが低い＝悪い」ではなく「ここから改善できる」というトーンで書く
- **—（対象外）の説明**: 対象外にした理由を1文で記述し、ユーザーが納得できるようにする
- **検出限界の明示**: ファイルベースの静的解析では検出できない項目（実際のワークフロー品質、暗黙知の有無等）があることを総評で正直に述べる
